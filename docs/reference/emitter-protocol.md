# Emitter Protocol Reference

External code generators (framework emitters) plug into `confiture generate scaffold` by implementing the `ConfitureEmitter` protocol. This page documents the contract, the supporting types, the semver promise, and three worked examples.

## Contract

```python
from typing import Protocol
from confiture.core.scaffold.emitter import EmittedFunction


class ConfitureEmitter(Protocol):
    def emit(self) -> list[EmittedFunction]: ...
```

`ConfitureEmitter` is a [`typing.Protocol`](https://docs.python.org/3/library/typing.html#typing.Protocol), so structural conformance is sufficient — you do not need to subclass anything. Any callable that, when invoked with no arguments, returns a `list[EmittedFunction]` will work with `--from module.path:callable_name`.

### The `EmittedFunction` dataclass

```python
@dataclass
class EmittedFunction:
    verb: str             # filename suffix (validated: ^[A-Za-z0-9][A-Za-z0-9_.\-]*$)
    subtree: Path         # target directory under schema_dir
    content: str          # SQL body (everything after the generated header)
    generated_marker: str # first line of the file header, conventionally a SQL comment
```

The orchestrator allocates the prefix automatically and writes:

```sql
{generated_marker}
-- Generated on: {ISO-8601 timestamp}
-- Do not edit — move to overrides/ to take ownership.

{content}
```

### Validation rules

The scaffold orchestrator enforces these constraints (failures raise `ValueError`, so `confiture generate scaffold` exits non-zero):

| Field | Rule |
|---|---|
| `verb` | Must match `^[A-Za-z0-9][A-Za-z0-9_.\-]*$` — letters, digits, underscore, hyphen, dot. **No path separators, no `..`.** This is a security boundary, not a style guide. See [#security](#security). |
| `subtree` | Must resolve inside `schema_dir`. Symlinks are followed; trying to escape via a symlink fails. |
| `content` | Free-form text. The orchestrator does not parse or validate the SQL. |
| `generated_marker` | Free-form text. By convention a SQL line comment so the file remains valid SQL. |

## Semver promise

The emitter protocol is **stable across 0.10.x and 1.x minor releases**.

| Change | Type |
|---|---|
| Adding a new optional field to `EmittedFunction` (with a default) | minor (back-compat) |
| Renaming or removing an existing field | major (breaking) |
| Tightening `verb`'s regex | minor with deprecation cycle; warnings for one release first |
| Loosening `verb`'s regex | minor (back-compat) |
| Adding a method to `ConfitureEmitter` | major (breaking; existing emitters would fail conformance) |

The `EmittedFunction` dataclass and `ConfitureEmitter` Protocol live at `confiture.core.scaffold.emitter`. Import only from that module — internal paths can move between releases.

## Worked example 1 — trivial single emitter

```python
# myproject/generators.py
from pathlib import Path
from confiture.core.scaffold.emitter import EmittedFunction


def emit_hello() -> list[EmittedFunction]:
    return [
        EmittedFunction(
            verb="hello_world",
            subtree=Path("db/schema/functions/sandbox"),
            content="""
CREATE OR REPLACE FUNCTION hello_world()
RETURNS text
LANGUAGE sql
AS $$
    SELECT 'hello, world'::text;
$$;
""".strip(),
            generated_marker="-- GENERATED by myproject v0.1",
        )
    ]
```

Run:

```bash
confiture generate scaffold --from myproject.generators:emit_hello
```

Result on disk:

```sql
-- db/schema/functions/sandbox/00001_hello_world.sql
-- GENERATED by myproject v0.1
-- Generated on: 2026-05-20T11:35:00+00:00
-- Do not edit — move to overrides/ to take ownership.

CREATE OR REPLACE FUNCTION hello_world()
RETURNS text
LANGUAGE sql
AS $$
    SELECT 'hello, world'::text;
$$;
```

## Worked example 2 — GraphQL CRUD-style emitter

A real emitter typically iterates over an introspected model and emits one function per (entity × verb) pair.

```python
# myproject/crud_emitter.py
from pathlib import Path
from confiture.core.scaffold.emitter import EmittedFunction

ENTITIES = ["item", "order", "customer"]
VERBS = ["create", "update", "delete", "fetch"]


def emit_crud() -> list[EmittedFunction]:
    out: list[EmittedFunction] = []
    for entity in ENTITIES:
        subtree = Path("db/schema/functions") / entity
        for verb in VERBS:
            out.append(
                EmittedFunction(
                    verb=f"{verb}_{entity}",
                    subtree=subtree,
                    content=_render(entity, verb),
                    generated_marker="-- GENERATED by myproject-crud v1.2",
                )
            )
    return out


def _render(entity: str, verb: str) -> str:
    # In real code: load a Jinja template, render with entity context.
    return f"""
CREATE OR REPLACE FUNCTION {verb}_{entity}(...)
RETURNS ...
LANGUAGE plpgsql
AS $$
    -- generated body
$$;
""".strip()
```

The orchestrator allocates `00001_create_item.sql`, `00002_update_item.sql`, etc. inside each entity subtree. Re-running `scaffold` is idempotent: existing files are not overwritten (next allocations pick up after the highest existing prefix).

## Worked example 3 — overrides

When a user wants to take permanent ownership of a generated file, they copy it into an overrides mirror directory:

```text
db/schema/functions/item/00001_create_item.sql     # generated
overrides/functions/item/00001_create_item.sql     # user-owned
```

```bash
confiture generate scaffold --from myproject.crud_emitter:emit_crud \
                            --overrides-dir overrides/
```

The orchestrator detects the override and reports `action="skip"` for that path. The user's version is left untouched; future regenerations of `00001_create_item.sql` never overwrite it.

The override matches by **relative path** under the schema dir. Moving the override file changes which generated path it matches.

## Security

Two boundaries are explicitly tested:

1. **`verb` cannot contain path separators or `..`.** Without this rule, a verb like `"../../../etc/passwd"` would let `Path.mkdir(parents=True)` and `write_text` escape the schema directory entirely. The pattern `^[A-Za-z0-9][A-Za-z0-9_.\-]*$` rules out the whole class.
2. **`subtree` is resolved (symlinks followed) and checked against `schema_dir`.** Symlinks inside the schema tree that point outside still fail validation.

Emitters are trusted Python code; if you allow an untrusted user to register an emitter, they already have arbitrary code execution. The boundaries above protect against accidental escapes by well-meaning generators.

## Design decisions

The following questions were resolved when finalizing the protocol in [#111](https://github.com/fraiseql/confiture/issues/111). They are not negotiable per-call; raise a new issue with concrete requirements if you need different behaviour.

### 1. Numbering scheme: decimal-5, fixed

`PrefixConfig.scheme` and `PrefixConfig.width` exist for backward compatibility with existing hex-numbered or 2-digit-wide schema trees, but new projects get decimal-5 (`00001`, `00002`, …). Configurability per project was rejected because:

- Hex prefixes don't sort correctly in graphical tools (`0001a` vs `00010`).
- Width tuning is solved by re-running `renumber` at scale, not by per-project config.
- One scheme keeps cross-project tooling simpler.

### 2. `alloc` is stateless

There is no reservation file, no advisory lock, no per-branch counter. Every `alloc` call re-reads the directory and returns the next free prefix. The consequence: parallel branches can collide on prefixes and need `renumber` after merge. We considered:

- Reservation files (rejected — adds an additional source of truth that drifts from filesystem state).
- Advisory locks (rejected — alloc is a filesystem operation, not a database one).
- Per-branch counters (rejected — branches don't have stable identifiers across reboots / rebases).

The "manual renumber after merge" cost is real but small; the simplicity of the stateless model carries.

### 3. Emitter interface: Python `Protocol` only for v1

A CLI/JSON contract (`--from-cmd 'some-binary | jq ...'`) was considered for cross-language emitters and deferred. No user has requested cross-language emitters; until one does, `Protocol`-only keeps the surface area smaller.

If you need cross-language: file an issue with the concrete external generator you want to plug in. The right shape is `--from-cmd 'cmd args'` emitting JSON to stdout, validated against a schema in `docs/reference/notification-payload-schema.json`-style.
