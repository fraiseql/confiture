# Example 07 — External Emitter

A minimal working example of a `ConfitureEmitter` plugged into `confiture generate scaffold`.

The emitter is **50 lines of Python**, emits **two functions** (`ping` and `echo_ping`), and demonstrates:

- How to implement the `emit()` callable.
- How `confiture generate scaffold --from <module>:<callable>` discovers and invokes it.
- What the generated files look like (header, prefix allocation, sort order).
- How the override mirror lets a user take permanent ownership of a generated file.

## Layout

```text
07-external-emitter/
├── emitter.py                                  # the emitter callable
├── confiture.toml                              # minimal config
├── db/
│   └── schema/
│       └── functions/
│           └── sandbox/                        # target directory (populated by scaffold)
└── README.md
```

## Run it

From this directory:

```bash
confiture generate scaffold --from emitter:emit --schema-dir db/schema
```

After the run, you'll see:

```text
db/schema/functions/sandbox/
├── 00001_ping.sql
└── 00002_echo_ping.sql
```

Each file starts with the generated header the orchestrator writes:

```sql
-- GENERATED by example-emitter v0.1.0
-- Generated on: 2026-05-20T11:39:00+00:00
-- Do not edit — move to overrides/ to take ownership.

CREATE OR REPLACE FUNCTION ping()
...
```

## Try the override mechanism

The override mechanism matches by **allocated path**, so the workflow that exercises it is **wipe-then-regen**: clear the generated directory, ensure any files you own live in `overrides/`, run scaffold.

```bash
# Put your version where the orchestrator will look for it.
mkdir -p overrides/functions/sandbox/
cp db/schema/functions/sandbox/00001_ping.sql overrides/functions/sandbox/
# Edit overrides/functions/sandbox/00001_ping.sql — make it your own.

# Wipe the previously generated files and re-run with --overrides-dir.
# The orchestrator will allocate 00001 first, see the override, and skip.
rm -f db/schema/functions/sandbox/*.sql
confiture generate scaffold --from emitter:emit \
                            --schema-dir db/schema \
                            --overrides-dir overrides/
```

Output:

```text
~ skip: db/schema/functions/sandbox/00001_ping.sql
✓ write: db/schema/functions/sandbox/00002_echo_ping.sql
```

The orchestrator allocated `00001_ping.sql`, saw a matching path under `overrides/`, and reported skip. The file in `db/schema/` was never written — you'll consume the override directly from the build step (`confiture build` concatenates DDL from `db/schema/` only by default; wire the overrides into your build by adding `overrides/` to `include_dirs` in your environment YAML).

**Caveat**: the mechanism is path-keyed, not verb-keyed. If you don't wipe before re-running, the allocator allocates a *new* prefix (`00003_ping.sql`) and the override won't match, so the file lands in `db/schema/` and you end up with both versions.

## What to learn from this

1. **The emitter is just a callable.** It can read your own schema, render Jinja, query an API — anything that produces a `list[EmittedFunction]`.
2. **`verb` is validated.** Try changing the `verb` to `"../escape"` and re-running scaffold — you'll get a clear `ValueError` instead of a path-traversal write. See [`docs/reference/emitter-protocol.md#security`](../../docs/reference/emitter-protocol.md#security).
3. **Cross-references are tracked.** If you later run `confiture generate renumber db/schema/functions/sandbox/00001_ping.sql db/schema/functions/sandbox/00099_ping.sql`, the call to `ping()` inside `echo_ping.sql` will be detected as a reference (not rewritten, since the stem didn't change).

## Cleaning up

```bash
rm -rf db/schema/functions/sandbox/*.sql overrides/
```
